# Terminology

## Project Names

- **Quince**: The overall local-first production suite containing Stager, LineRecorder, and Cuemaster. It is also the intended wake word for future voice-driven workflows.
- **Stager**: The existing Python CLI/build tool in this repository. Stager generates role scripts, audio plays/audiobooks, verification artifacts, and Playbooks.
- **ScriptWright**: The Stager-side ingestion tool that converts supported draft script inputs, including the current `play.txt` format and draft `production.md`, into canonical locked `production.md`.
- **LineRecorder**: The actor-facing recording tool. LineRecorder consumes Stager-generated Recording Requests and exports segment-aware recording packages for Stager to import.
- **Cuemaster**: The mobile rehearsal app for actors learning lines. Cuemaster consumes Playbooks generated by Stager.
- **Playbook**: The portable package generated by Stager and consumed by Cuemaster. A Playbook contains `manifest.json` plus audio assets.
- **Production script**: The human-editable canonical script artifact maintained during rehearsal. The planned filename is `plays/<play_id>/production.md`; draft versions may be idless or provisionally id-bearing, while locked versions have stable production ids.
- **Production id**: A permanent, concise, human-readable identifier for an addressable script unit, such as `3.2-15`, `3.2-15a`, or `3.2-15:s1`.

## Core Domain Terms

- **Play**: Parsed source text and metadata represented by Stager's `Play` model.
- **Role**: A speaking part in a play. Special roles include `_NARRATOR`, `_CALLER`, and `_ANNOUNCER`.
- **Part**: A top-level section of a play, represented by `Part` and `BlockId.part_id`.
- **Block**: A parsed source-text block with a stable `BlockId`, such as `0.2`.
- **Segment**: A speakable or narratable unit within a block with a stable `SegmentId`, such as `0_2_1`.
- **Cue**: Text and required audio that prompts an actor before their expected response in a Playbook.
- **Line**: User-facing wording for an actor's speakable unit in LineRecorder or a Cuemaster rehearsal item. In data contracts and code, prefer `segment` when referring to Stager audio identity.
- **Recording item**: A LineRecorder prompt shown to an actor for recording. A recording item usually maps to exactly one Stager `SegmentId`; if a displayed source line contains multiple speakable segments, LineRecorder should record them as separate segment-backed items.
- **Recording Request**: A Stager- or Cuemaster-generated work order for LineRecorder. It may request a full role recording or a constrained set of changed/problem segments.
- **Response**: The actor's expected spoken line or segment group.
- **Manifest**: `manifest.json` inside a Playbook. It is the structured contract Cuemaster consumes.

## Naming Rules

- Use **Quince** for the suite, not as a replacement for individual tool names.
- Use **Stager CLI** when referring to `./main` and `src/stager/cli/build.py` behavior.
- Use **ScriptWright** for the ingestion/hardening tool that creates or locks `production.md`; use **Stager** for builds that consume locked `production.md`.
- Use **LineRecorder** when referring to actor-side recording and recording package import/export workflows.
- Use **Cuemaster** when referring to the mobile app or its runtime behavior.
- Use **Playbook** for generated portable packages, not `scriptpack`, `app package`, or `play package`.
- Use **manifest** for the JSON contract inside a Playbook.
- Use **Recording Request** for Stager- or Cuemaster-to-LineRecorder work orders and **recording package** for LineRecorder-to-Stager audio exports.
- Keep **line** as actor-facing UI language; use **segment** for manifest fields, filenames, Stager IDs, and implementation details.
- Treat Playbooks as strict artifacts: required cue and response audio must exist for every rehearsable non-meta role line.
- Keep `Androcles` only when referring to the sample/source play, not the tool.
